## What it is & for whom

**Quaero** (Latin: "I seek / I ask") — a **cross-platform desktop** app
(macOS, Windows, Linux) to chat over PDFs via **RAG**, with **verifiable
citations** (document + page + snippet resolved from the database, never
generated by the LLM). Runs **100% locally**, ~zero cost, no deploy. Built with
**Electron** (Node.js main process + React 19 renderer).

**Personas:**

| Persona | Profile | Core need |
|---|---|---|
| **Alex** (primary) | Legal / Compliance Analyst | Review contracts / tenders / regulations (30–100 pp); find clauses (termination, liability, confidentiality); **cite the exact page**. Currently uses Ctrl+F and misses cross-references. |
| **Pat** (secondary) | Graduate Researcher | Works with papers / reports (50–200 pp); asks comparative questions across documents; needs verifiable citation for literature-review notes. |

## In Scope

### PDF Upload
- Native file dialog **or** drag-and-drop, for one or more PDFs.
- Cap: **200 MB** per file (configurable in settings).
- Asynchronous processing: up to **1000 pages / 200 MB** per file; above → clear error.

### Processing (async, on upload)
1. **Per-page** text extraction (`pdfjs-dist`) — required for citation.
2. **Chunking**: character sliding window, 512 chars, 64 overlap, sentence-boundary preference (`.`, `!`, `?`, line break); minimum chunk merges pages < 64 chars; page metadata preserved.
3. **Embedding** of each chunk via Ollama (`bge-m3`, 1024 dims).
4. **Storage** in `document_chunks.embedding` (sqlite-vec, `float32(1024)`).
5. Returns `document ready` (or `failed` with `error_code`).

### Document Chat (RAG)
- Natural-language questions about the conversation's documents.
- Flow: embed question → vector search (cosine) over the associated documents' chunks → top-K → prompt builder (answer **only** from context) → `LlmProvider` (GLM-5.1 remote **or** Ollama local) → persist answer + citations.
- **Refusal** when no chunk passes the similarity threshold (cosine > 0.5, configurable): do not call the LLM; return a pre-defined message (token savings + anti-hallucination).

### Source Citation (persisted)
- Every answer carries: **document name + page + snippet**.
- Saved to `message_citations` → **survives app restart**.

### Conversation History
- List previous conversations; continue an existing one.
- A conversation references **N** documents.

### Frontend (React 19 + Shadcn/ui + Tailwind)
- **Documents Screen**: drop-zone + native dialog; table with status badges (processing/ready/failed); rename and delete (cascade).
- **Chat Screen**: user/assistant messages; Markdown with expandable citations; sidebar of the conversation's documents; suggested prompts on empty state.
- **Conversations Screen**: conversation list; create new + associate documents; continue a previous one.
- **Theme**: light/dark/system, persisted to localStorage, **FOUC-free** boot.

### Cross-cutting
- **No auth, single-user.**
- LLM provider configured via settings UI (no file editing).
- Context isolation **on**, `nodeIntegration` **off**, all main-process access mediated through a typed `contextBridge`.
- Cross-platform distribution: `.dmg` / `.msi` / `.AppImage` via `electron-builder`.

## Out of Scope

> Do not implement in the MVP. Everything below is "future" (PRD §6) or
> "out of MVP" (`docs/STEERING.md`). Confirmed out of scope:

| Item | Reason | Reopen when |
|---|---|---|
| **MCP server** (expose tools for Claude Desktop, etc.) | Future (PRD §6) | Post-v1, explicit demand |
| **JWT / multi-user auth + isolation** | No auth, single-user, focus on the RAG core | Real multi-user need arises |
| **OCR for scanned PDFs** | Future (PRD §6); MVP only processes text PDFs | Validation shows demand |
| **PDF comparison / table extraction** | Future (PRD §6) | Post-v1 |
| **Export (Markdown/PDF/DOCX) / structured auto-summarization** | Future (PRD §6) | Post-v1 |
| **In-process embeddings (ONNX)** — remove Ollama dependency | Future (PRD §6); v1 uses Ollama `bge-m3` | After evaluating cost/quality |
| **Configurable / multi-model embeddings (different dims)** | Future (PRD §6) | Post-v1 |
| **Auto-update via `electron-updater`** | Future (PRD §6); v1 = manual distribution | Post-v1 (OQ-3: code signing) |
| **Token-based chunking** | v1 uses **characters** (512); token-based is a future iteration (OQ-4) | After Bolt 2, if recall is poor |

## Open Questions (known uncertainties)

> Mapped 1:1 from PRD §18. These are **known** uncertainties — they don't need to
> be rediscovered by the AI; they need to be **resolved** by the stated deadlines.

| # | Question | Owner | Deadline | Default if unresolved |
|---|---|---|---|---|
| **OQ-1** | `pdfjs-dist` extraction quality on edge-case PDFs (rare tables/columns). Needs validation. | Developer | End of Bolt 1 | Stick with `pdfjs-dist` (standard, robust); document limitations + tune rendering flags if needed (e.g. `disableFontFace`, `useSystemFonts`). |
| **OQ-2** | Should the app **auto-detect** Ollama and **pull `bge-m3`** automatically, **or** be documented as a prerequisite? | Stakeholder | Before Bolt 1 spec approval | Document as prerequisite in README + health-check error message. |
| **OQ-3** | **Code signing** (Apple notarization, Windows cert) for distribution. | Stakeholder | Post-v1 | **Unsigned** builds initially. |
| **OQ-4** | Chunk-size tuning: **character** vs **token** splitting. Evaluate after Bolt 2. | Developer | End of Bolt 2 | **Character (512 chars)** for v1. |
| **OQ-5** | Conversation history window: how many past messages to include in the prompt? | Developer | During Bolt 2 implementation | **Default 10 messages**, configurable via settings. |

### Other technical uncertainties (monitored risks — PRD §17)
- Maturity of the `sqlite-vec` Node bindings (npm `sqlite-vec`) — mitigation: pin exact version + load extension at startup.
- `pdfjs-dist` on odd layouts — mitigation: validate with 5 diverse PDFs in Bolt 1.
- Ollama as a hard external dependency — mitigation: health check on boot + clear install message.
- LLM hallucination despite grounding prompt — mitigation: strict prompt + refusal without context + citation validation (every citation resolves to a real chunk in the DB).
- Single-threaded embedding bottleneck — mitigation: `worker_threads`, configurable concurrency (default 8).

## Success Criteria (summary — PRD §14)

| Metric | Target |
|---|---|
| Chat latency | ≤ 5 s (P95) for docs ≤ 100 pp |
| Processing failure rate | ≤ 1% for valid PDFs within limits |
| Citation accuracy | ≥ 95% (correct doc + page) on benchmark |
| RAG Recall@5 | ≥ 90% on benchmark |
| App cold start | ≤ 5 s to interactive DocumentsView |
